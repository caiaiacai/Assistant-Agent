# SATH 软件设计说明书

> **软件名称**：SATH — 感知型智能待办中枢（Sensing Agentic Todo Hub）
> **版本**：V1.0
> **开发环境**：Tauri 2.0 + Rust + React 18 + TypeScript + SQLite
> **运行平台**：macOS（优先）、未来兼容 Windows / Linux

---

## 1. 软件概述

### 1.1 开发背景与目的

在个人知识工作高度碎片化的背景下，传统待办事项软件只解决"记录"问题，无法主动协助用户完成信息调研、情境感知和任务准备。用户经常面临以下痛点：

1. **想法散落**：灵感、任务、调研需求混杂在脑海里，没有统一捕获渠道；
2. **被动等待**：写下"下午 3 点开会"这样的提醒后，软件只会到点响铃，不会主动准备会议资料、路线、天气；
3. **多源割裂**：想法可能来自电脑、手机微信、Obsidian 笔记等多渠道，难以统一归集；
4. **缺乏感知**：软件不知道用户"现在在做什么"，无法结合情境给出建议。

SATH 旨在成为一个 **能主动思考、能感知环境、能跨渠道捕获** 的个人智能待办中枢，让用户只管"说出一个念头"，软件自动完成分类、规划、调研、提醒全流程。

### 1.2 主要功能

1. **一键捕获**：全局快捷键 `⌘⇧Space` 呼出主窗口，敲一行字即可入库；
2. **智能分类**：基于大语言模型（LLM）的 Planner 自动识别意图类型（research / task / note / reminder）；
3. **多 Agent 流水线**：Planner 生成包含多个 Agent 的执行 DAG，Executor 并行调度；
4. **情境感知**：集成系统原生定位（CoreLocation）、日期时间等情境信息；
5. **自然语言时间解析**：本地正则解析"下午 3 点"、"10 分钟后"等时间表达；
6. **跨渠道接入**：支持微信消息、Obsidian 笔记作为输入源；
7. **风险分级工具**：按用户设定的风险等级（保守 / 标准 / 大胆）过滤 agent 可用工具；
8. **可插拔 Agent 库**：从 Markdown 文件加载 agent 定义，支持自定义扩展；
9. **提醒调度**：到期自动推送系统通知。

### 1.3 技术特色

- **Tauri 2.0 原生壳**：产物体积小（< 20 MB），性能接近原生，跨平台；
- **Rust 后端**：内存安全、高并发、SQLite 本地持久化；
- **React 前端**：组件化、响应式 UI，符合现代 Web 开发规范；
- **多 Agent DAG 调度器**：支持 `depends_on` 依赖声明，独立步骤并行执行；
- **上下文压缩**：长对话自动做 LLM 摘要，节省 token；
- **隐私优先**：除 LLM 调用外，全部数据本地处理。

### 2.2 数据流

**典型流程：用户敲入"下午 3 点开会"**

```
① 用户在 Main.tsx 输入框回车
② 前端 invoke("create_todo", { content })
③ commands::create_todo 写入 SQLite，返回 Todo
④ reminder::parse_due_at 本地解析 "下午3点" → 得到 ISO 时间戳存入 due_at
⑤ planner::plan_one 异步启动（spawn）：
     a. 加载 Persona、Context、Agent Library
     b. 注入当前时间 + 位置信息到 prompt
     c. 调 LLM 生成 Plan（types / pipeline / due_at / ...）
     d. 本地 parser 结果覆盖 LLM 的 due_at（防止时间漂移）
     e. 写回 todos 表
⑥ executor::run_pipeline 按 DAG 调度各 Agent：
     a. depends_on 为空的 step 并行启动
     b. 每个 step 内部走 LLM + 工具调用循环
     c. 上下文超长时触发 maybe_compress_context
     d. 发出 sath://pipeline-progress 事件
⑦ 前端每 2.5 秒 list_todos 轮询 + 监听进度事件，实时刷新 UI
⑧ reminder::start_scheduler 每 20 秒扫描到期 todo，发出 sath://reminder 事件
⑨ 前端收到后用 Web Notification API 弹系统通知
```

## 3. 模块详细设计

### 3.1 Planner（规划器）— `planner.rs`

**职责**：接收用户原始输入，输出结构化的执行计划。

**核心方法**：
- `plan_one(db, todo_id, input, context_summary)`：主入口；
- `select_categories(...)`：两阶段选择第一阶段，当 Agent 库 > 12 个时先挑类目；
- `persona_and_context(...)`：组装 prompt 中的身份/时间/位置上下文。

**Prompt 设计哲学**：要求 LLM 执行"预判三步"——过去（用户从哪里来）、现在（他真正想要什么）、未来（他拿到结果后要做什么）。

**输出 Schema**：
```json
{
  "types": ["research", "task", "note", "reminder", "unknown"],
  "title": "<15 字内>",
  "tags": ["<标签>"],
  "due_at": "<ISO8601 或 null>",
  "confidence": 0.0-1.0,
  "reasoning": "<三步推演>",
  "pipeline": [
    {
      "agent": "<名称>",
      "display_name": "<中文短标签>",
      "brief": "<做什么>",
      "success_criteria": "<怎样算完成>",
      "extra_tools": [],
      "depends_on": [0, 1]
    }
  ]
}
```

### 3.2 Executor（执行器）— `executor.rs`

**职责**：按 Plan 的 pipeline DAG 调度 Agent，每个 Agent 在 LLM + 工具调用循环中完成任务。

**DAG 调度算法**：
1. 维护 `in_flight: HashSet<usize>` 和 `done: HashSet<usize>`；
2. 遍历所有 step，若其 `depends_on` 全部在 `done` 中且自己不在 `in_flight` / `done`，则 `tokio::spawn` 启动；
3. 结果通过 `Arc<Mutex<Vec<Option<Result>>>>` 收集；
4. 所有 step 完成后汇总。

**单 step 循环**：
```
初始化 messages（system + initial user）
for turn in 1..max_turns:
    if turn > 4 && messages.len() > 8:
        maybe_compress_context()  // 早期上下文压缩
    if remaining_turns <= 2:
        push 软性提示"立刻 final_answer"
    raw = llm::chat(messages)
    call = parse_tool_call(raw)
    if 解析失败 && raw 不像 JSON:
        视为隐式 final_answer
    if call.action == "final_answer":
        break
    if 连续两次相同调用:
        abort (原地打转保护)
    执行工具 → 拼接 tool_result 到 messages
```

**工具白名单**：`web_search` / `web_fetch` / `fs_list` / `fs_read` / `fs_write` / `file_download` / `shell_run`。

### 3.3 Agent 库 — `agent_library.rs`

Agent 定义写在 Markdown 文件里，支持 frontmatter 配置：
```yaml
---
name: competitor-watch
category: research
description: 竞品动态追踪
default_tools: [web_search, web_fetch]
risk: low
---

你是一个专门做竞品分析的 agent...
```

`bootstrap_defaults()` 首次启动时把内置 agent 写入 `app_data_dir/agents/`。`load_all()` 启动时扫描加载。

### 3.4 风险分级 — `risk.rs`

```
Conservative → 只允许 low 风险工具（web_search, web_fetch, fs_read, fs_list）
Standard     → + med 风险工具（fs_write, file_download）
Bold         → + high 风险工具（shell_run）
```

`filter_tools(tools, level)` 在每次 Executor 分派工具时做一次过滤。

### 3.5 数据库 — `db.rs`

使用 `rusqlite`（bundled feature，避免外部依赖）。主要表：

```sql
CREATE TABLE todos (
  id TEXT PRIMARY KEY,
  content TEXT NOT NULL,
  title TEXT,
  notes TEXT,
  agent_output TEXT,
  types TEXT,            -- JSON array
  confidence REAL,
  needs_review INTEGER,
  status TEXT NOT NULL DEFAULT 'open',
  priority INTEGER DEFAULT 0,
  due_at TEXT,
  tags TEXT,             -- JSON array
  context_snapshot TEXT, -- JSON
  agent_status TEXT DEFAULT 'idle',
  agent_error TEXT,
  classify_status TEXT DEFAULT 'pending',
  plan TEXT,             -- JSON
  origin TEXT DEFAULT 'local',
  source_id TEXT,
  last_synced_hash TEXT,
  last_synced_at TEXT,
  reminder_fired INTEGER NOT NULL DEFAULT 0,
  created_at TEXT NOT NULL,
  updated_at TEXT NOT NULL
);

CREATE TABLE settings (
  key TEXT PRIMARY KEY,
  value TEXT NOT NULL
);
```

启动时执行幂等的 `ALTER TABLE ... ADD COLUMN` 迁移，保证向前兼容。

### 3.6 微信网关 — `gateway/wechat.rs`

基于 vendored 的 `wechatbot` 库（已 fork 并修 bug）：
- 扫码登录 → 发出 `sath://wechat-qr` 事件给前端显示二维码；
- 收到文本消息 → 创建 todo（`origin: "wechat"`）；
- Agent 跑完后 → LLM 二次总结关键信息（`summarize_for_wechat`）→ 回发给用户；
- `strip_think()` 过滤掉 LLM 输出中的 `<think>...</think>` 内容推理块。

### 3.7 Obsidian 网关 — `gateway/obsidian.rs`

- 用户指定 vault 路径和子文件夹；
- 扫描 `.md` 文件的 YAML frontmatter 中 `sath: true` 字段 → 视为任务；
- 双向同步：SATH 里写入 → Obsidian 文件更新；Obsidian 文件改动 → `last_synced_hash` 对比 → 更新 DB；
- `obsidian_full_sync()` 手动全量。

### 3.8 提醒调度器 — `reminder.rs`

**本地时间解析**：正则匹配 `N分钟后 / N小时后 / 今天 / 明天 / 后天 / 凌晨 / 早上 / 上午 / 中午 / 下午 / 傍晚 / 晚上 / H点[半/M分] / H:M`，总能解析成 ISO8601。

**Scheduler**：`tauri::async_runtime::spawn` 起一个 tick 任务，每 20 秒 `SELECT ... WHERE due_at <= now AND NOT reminder_fired` → 发出 `sath://reminder` 事件 → 前端调用 Web Notification API。

### 3.9 原生定位 — `location_macos.rs`

通过 `objc2-core-location` crate 直调 macOS CoreLocation 框架：
1. 独立线程起 `CLLocationManager`；
2. 请求 `requestWhenInUseAuthorization`；
3. 在线程里泵 `NSRunLoop` 等待 `location` 属性被填充；
4. 通过 mpsc channel 返回 `(lat, lng)` 给 Tokio 异步上下文。

打包后的 `.app` 会从 `src-tauri/Info.plist` 继承 `NSLocationUsageDescription` 字段，首次运行时弹出系统权限对话框。

### 3.10 设置存储 — `settings.rs`

统一 KV 风格，`get_json<T>` / `set_json<T>` 把任意 serde 结构存入 `settings` 表。主要配置：

- `llm_config`：base_url / api_key / model
- `agent_config`：max_turns / timeout / search_provider / amap_key / ...
- `persona`：用户角色、当前关注、典型工作、偏好
- `context_config`：情境采集开关
- `risk_level`：Conservative / Standard / Bold
- `obsidian_config`、`wechat_config`、`lark_config`
- `location`：LocationInfo (lat, lng, city, address, updated_at)

### 3.11 前端页面结构

- **`Main.tsx`**：主界面，捕获输入框 + 过滤栏 + TODO 列表 + 详情展开；
- **`Settings.tsx`**：Persona / LLM / Agent / Context / Risk / Obsidian / WeChat / Location 全部设置；
- **`Onboarding.tsx`**：首次启动引导；
- **`components/Wawa.tsx`**：视觉识别系统（VI）的蛙 IP、莲叶、宝石、水滴 SVG 组件；
- **`api.ts`**：所有 Tauri invoke 调用的类型化包装；
- **`types.ts`**：前后端共享的 TS 类型定义。

### 3.12 视觉识别系统（VI）

主色蓝 `#2B4FD8`（Wawa 蛙本体）、莲叶绿 `#7DC96F`、宝石蓝 `#1E3A9E`、水滴浅蓝 `#9BC4F0`。主窗口固定 460 px 宽的竖屏手机布局，居中显示，Mac 桌面上呈现为挂在角落的小工具风格。

## 4. 关键技术决策

### 4.1 为什么选 Tauri 而非 Electron？

- **产物体积**：Tauri 用系统 WebView（macOS 上是 WKWebView），产物 < 20 MB，Electron 动辄 100+ MB；
- **性能**：Rust 后端 + 原生 WebView，内存占用显著更低；
- **安全**：Rust 的内存安全 + Tauri 的能力白名单机制。

### 4.2 为什么选 SQLite 而非 JSON 文件？

- 事务保证：多个字段一起更新不会半途失败；
- 查询灵活：过滤、排序、全文搜索都能走 SQL；
- 未来扩展：全文索引、多用户支持、同步框架都依赖关系型结构。

### 4.3 为什么要两阶段 Planner？

当 Agent 库数量 > 12 时，把全部 agent 塞 prompt 会严重增加 token 消耗，也会让 LLM 注意力分散。先用轻量 prompt 选 ≤3 个类目，再在这些类目里展开具体 agent 列表，既省 token 又提升选择准确率。

### 4.4 为什么要 DAG 而非链式 Pipeline？

研究类任务经常有多个**独立调研方向**（例如"竞品分析 + 价格调研 + 技术栈调研"），如果串行跑要等很久。通过 `depends_on` 声明依赖，让 Executor 把独立 step 并行起来，在 LLM 推理是瓶颈的场景下几乎线性加速。

### 4.5 为什么要上下文压缩？

长任务的 Agent 循环可能跑 10+ 轮，messages 数组不停膨胀，触及 LLM 上下文上限。`maybe_compress_context` 在 turn > 4 && messages > 8 时，把中间的 assistant+tool_result 对用一次 LLM 调用压缩成一条 "早期调研结论" 摘要，保留 system + 初始 user + 最近 2 对，token 节省 40-70%。

## 5. 运行环境

- **操作系统**：macOS 11+（优先）、Windows 10+、主流 Linux（理论支持）
- **硬件**：无特殊要求，支持 Apple Silicon 和 Intel
- **网络**：联网以使用 LLM 和网络搜索功能，离线状态下仅失去 LLM 相关能力
- **依赖服务**：用户需自备 OpenAI 兼容 LLM API（支持 OpenAI / Anthropic / DeepSeek / 通义千问等）

---

## 6. 数据安全与隐私

1. **本地优先**：所有 todo、设置、情境数据存储在用户本地 SQLite 数据库；
2. **LLM 外发**：仅在调用 LLM 时把必要内容（任务描述 + 上下文）发送给用户配置的 LLM 服务商；
3. **敏感信息脱敏**：`redact.rs` 在 prompt 组装前移除邮箱、电话、身份证号等模式；
4. **微信凭据**：仅存储用于 wechatbot 库的登录 token，不上传第三方；
5. **地理位置**：CoreLocation 授权后仅本地使用，若配置高德反查，才会以坐标换取城市名，不涉及身份。
